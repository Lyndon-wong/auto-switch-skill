## 1. 项目概述

### 1.1 项目背景

在 OpenClaw 多智能体系统中，每个 Agent 绑定固定的 LLM 模型。然而在实际使用中，任务的复杂度和类型差异巨大——从简单的问答、格式转换，到复杂的逻辑推理、长文档分析、代码生成。固定绑定单一模型会导致两个核心问题：

1. **能力不足**：轻量模型（如 Qwen-7B）面对复杂推理任务时质量下降
2. **资源浪费**：高性能模型（如 GPT-4）处理简单任务时造成不必要的 Token 消耗和费用

### 1.2 项目目标

设计并实现一个 OpenClaw Skill 模块，使 Agent 能够根据任务类型和难度**自动切换**底层 LLM 模型，在保证任务质量的同时优化成本。

### 1.3 核心价值

| 维度 | 描述 |
| :--- | :--- |
| **智能化** | Agent 具备"自知之明"，能识别自身能力边界 |
| **经济性** | 简单任务用小模型，复杂任务用大模型，降低总体成本 |
| **无缝性** | 模型切换对用户透明，上下文信息不丢失 |
| **可扩展** | 易于接入新模型、新的任务分类器 |

### 2.2 [F-001] 模型切换模块（Model Switcher）

#### 2.2.1 需求描述

根据用户输入的任务类型，通过**阻尼式切换条件**判断是否需要切换模型，并执行实际的模型切换操作。整个流程包含三个步骤：任务类型评估 → 切换条件判定 → 执行切换。

#### 2.2.2 步骤一：任务类型评估

##### 评估方式：模型自评估（Self-Evaluation）

通过在 System Prompt 中嵌入分类指令，让当前模型结合上下文**自行判断任务类型标签**。模型只负责输出任务类型，不负责评估难度分数——难度由任务类型通过静态配置表映射得出。

**选择理由**：
- 模型理解自身上下文，类型判断准确率最高
- 不引入额外 API 调用延迟
- 额外 Token 消耗极小（约 200-350 tokens/次，占比 <5%）
- 实现简单，仅需在 System Prompt 中追加分类指令

**输出格式约定**：模型在 `thought` / `thinking` 字段中以 `[TASK_TYPE: XXX]` 格式输出任务类型标签，由 Skill 解析提取。

##### 任务类型分类

| 类型标识 | 任务类型 | 典型场景 |
| :--- | :--- | :--- |
| `CHAT` | 日常对话 | 闲聊、问候、简单问答 |
| `QA` | 知识问答 | 事实查询、概念解释 |
| `SUMMARIZE` | 摘要生成 | 文档摘要、会议纪要 |
| `TRANSLATE` | 翻译 | 多语言互译 |
| `CODE_SIMPLE` | 简单编程 | 函数编写、Bug 修复 |
| `CODE_COMPLEX` | 复杂编程 | 架构设计、大规模重构 |
| `REASONING` | 逻辑推理 | 数学证明、因果分析 |
| `ANALYSIS` | 深度分析 | 长文档分析、数据解读 |
| `CREATIVE` | 创意生成 | 文案创作、方案设计 |
| `PLAN` | 执行计划 | 任务规划、实施方案制定、项目拆解 |
| `MULTI_STEP` | 多步骤任务 | 工作流编排、复合任务 |

##### 模型任务适配度矩阵（Routing Matrix）

> 任务类型**不映射到固定层级**，而是通过**模型任务适配度矩阵**直接查询每个可用模型对该任务的适配度权重（0-100）。
> 系统在安装时自动读取可用模型列表，与内置画像库 `model_profiles.yaml` 模糊匹配，生成用户专属的 `routing_matrix.yaml`。
> 用户可通过 `/ms router` 命令查看和修改该矩阵。

**安装时自动生成流程**：
1. 读取 OpenClaw `openclaw.json` 中已注册的可用模型列表
2. 将每个模型 ID 与内置画像库 `model_profiles.yaml` 模糊匹配
3. 匹配成功的模型继承画像权重，未匹配的模型所有任务权重默认为 3
4. 生成 `routing_matrix.yaml` 文件，并输出摘要表格供用户确认
5. 用户可通过 `/ms set <模型> <任务缩写> <适配度>` 随时修改任意权重

**路由逻辑**：对于任务类型 T，在可用模型中按 `routing_matrix[model][T]` 降序排列，选取权重最高的模型作为推荐目标。

**适配度权重参考标准**：

| 权重范围 | 含义 |
| :--- | :--- |
| 90-100 | 最佳适配（首选模型） |
| 70-89 | 高度适配（很好完成，但非最优） |
| 50-69 | 基本适配（可完成，质量/效率非最佳） |
| 30-49 | 勉强适配（能力边界，仅在无更优选项时使用） |
| 0-29 | 不推荐（基本不适合此任务） |


#### 2.2.3 步骤二：切换条件设计

切换决策**不单纯依赖任务类型**，而是采用**多信号融合的阻尼系统**，防止模型频繁切换。核心包含三个机制：适配度矩阵路由、切换势能累积、上下文惯性。

##### （一）任务类型 → 推荐模型选取（适配度矩阵路由）

模型自评估只输出任务类型标签，推荐模型由**适配度矩阵**直接查询得出（不让模型参与难度评判）：

**选取流程**：
1. 获取当前任务类型标签（如 `CODE_COMPLEX`）
2. 从 `routing_matrix.yaml` 中查询所有可用模型对该任务类型的权重
3. 按权重降序排列，选取**权重最高**的模型为推荐目标
4. 计算推荐模型权重与当前模型权重的**差值** → 用于判定切换方向

**切换方向判定**（基于权重差值）：

| 条件 | 方向 | 说明 |
| :--- | :--- | :--- |
| `推荐权重 - 当前权重 > weight_diff_threshold` | **升级** | 推荐模型明显更适合当前任务 |
| `当前权重 - 推荐权重 > weight_diff_threshold` | **降级** | 当前模型性能过剩，可节省成本 |
| 差值绝对值 ≤ `weight_diff_threshold` | **不触发** | 两者能力接近，不累积势能 |

> ⚠️ 适配度矩阵为安装时自动生成的默认配置，可通过 `/ms set` 修改，也可在 Agent 级别覆盖。

##### （二）切换势能机制（Switch Potential）

引入"切换势能"概念，模拟物理阻尼效果。**不是一次命中就切换，而是持续累积势能，只有越过阈值才真正触发切换**。

势能更新规则（每轮对话执行一次）：
- 如果推荐模型 ≠ 当前模型 且 权重差值超过阈值：`势能 += 方向增量`
- 如果推荐模型 == 当前模型，或权重差值未超过阈值：`势能 = max(势能 - 衰减值, 0)`
- 当 `势能 >= 方向阈值` 时 → 触发切换，势能归零

**方向不对称阈值**——升级敏感，降级保守：

| 切换方向 | 阈值 | 每轮增量 | 衰减值 | 约需连续轮数 | 设计原因 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **升级**（推荐权重高于当前） | 50 | +40 | 30 | ~2 轮 | 能力不足直接影响输出质量，需快速响应 |
| **降级**（推荐权重低于当前） | 80 | +20 | 30 | ~4-5 轮 | 大模型处理小任务不出错，只是浪费成本，不紧急 |

##### （三）上下文惯性（Context Inertia）

防止在连贯的多轮任务中因偶尔插入简单消息（如"好的""谢谢"）而触发误切换。

**惯性生效条件**：
- 最近 N 轮属于同一主题/任务的连续对话
- 未出现明确的话题转换信号

**惯性效果**：
- 切换阈值乘以 **×1.5 放大倍数**（更难触发切换）

**话题转换信号**（触发惯性重置）：
- 用户消息中出现转折语句（如"新的需求"、"另一个问题"、"换个话题"、"接下来"）
- 上一轮为明确的任务完成信号（如"搞定了"、"可以了"）
- 超过设定时间间隔未对话（默认 30 分钟）

##### （四）完整决策流程

```
用户消息进入
    │
    ▼
[冷却期检查] ← 上次切换后是否已过冷却轮数？
    │
    ├── 冷却中 → 跳过评估，使用当前模型
    │
    └── 已过冷却期
         │
         ▼
    [模型自评估] → 输出任务类型标签（如 CODE_COMPLEX）
         │
         ▼
    [适配度矩阵查询] → 该任务类型最优模型 + 权重
         │
         ▼
    权重差值是否超过 weight_diff_threshold？
         │
         ├── 否（差值过小） → 势能衰减，不切换
         │
         └── 是 → 判断切换方向（升级/降级）
                   │
                   ▼
              势能 += 方向增量
                   │
                   ▼
              [上下文惯性检查]
                   │
                   ├── 惯性高 → 阈值 ×1.5
                   └── 惯性低 → 使用原始阈值
                        │
                        ▼
                  势能 >= 最终阈值？
                        │
                        ├── 否 → 不切换
                        └── 是 → ✅ 执行切换
                                   势能归零
                                   冷却计时开始
```

##### （五）效果举例

假设用户当前使用 **Claude-Opus-4**（CODE_COMPLEX 权重 96）做复杂编程任务：

| 轮次 | 用户消息 | 类型判断 | 推荐模型(权重) | 当前模型(权重) | 差值 | 势能 | 切换？ | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | "帮我重构这个模块" | CODE_COMPLEX | Claude-Opus(96) | Claude-Opus(96) | 0 | 0 | ❌ | 一致，衰减 |
| 2 | "好的，谢谢" | CHAT | GPT-5.4(95) | Claude-Opus(90) | +5 | 0 | ❌ | 差值<15，不累积 |
| 3 | "函数命名改一下" | CODE_SIMPLE | GPT-5.4(90) | Claude-Opus(86) | +4 | 0 | ❌ | 差值<15，不累积 |
| 4 | "看看这段逻辑对不对" | REASONING | o3(98) | Claude-Opus(92) | +6 | 0 | ❌ | 差值<15，不累积 |
| 5 | "搞定了" | CHAT | GPT-5.4(95) | Claude-Opus(90) | +5 | 0 | ❌ | 差值<15，不累积 |

> 💡 **观察**：由于旗舰模型在大多数任务上的权重都很高，权重差值很难超过阈值，这自然体现了"大模型不会被轻易降级"的设计意图。降级主要发生在从推理专精模型（如 o3，CHAT 权重仅 42）切换到通用对话场景时。

#### 2.2.4 步骤三：执行切换

基于 OpenClaw 的 `session_status(model="目标模型")` API 实现模型切换。

##### 切换约束（兜底规则）

- **成本上限**：单次会话内模型切换总次数限制（默认 30 次）
- **用户确认**：升级到旗舰模型时可选择是否需要用户确认
- **回退机制**：切换失败时自动回退到上一个可用模型

##### 接口定义

```python
class TaskEvaluation:
    task_type: str               # 任务类型标识（如 CODE_COMPLEX）
    recommended_model: str       # 推荐的具体模型标识（从适配度矩阵中选取）
    recommended_weight: int      # 推荐模型在该任务上的权重（0-100）
    current_weight: int          # 当前模型在该任务上的权重（0-100）
```

### 2.3 [F-002] 上下文管理模块（Context Manager）

> **设计理念**：每次模型切换（switch）同时完成三件事——**保留**关键信息、**压缩**任务进展、**丢弃**冗余内容。切换即瘦身，瘦身即切换。
>
> **设计来源整合**：ClawHub Context Pruner（分类裁剪）、DR Context Pipeline（确定性管道）、分层摘要 DAG（学术实践）、OpenClaw 三层记忆架构（2026 社区标准）。

#### 2.3.1 需求描述

在模型切换过程中，通过**分层记忆架构**和**智能提取管道**，实现上下文的精准保留与高效瘦身。目标：
- 关键信息零丢失（决策、架构、约束、错误教训）
- 长距离历史信息可追溯（通过摘要链 + PIN 机制）
- 每次切换后 Token 利用率提升 40%+（丢弃冗余内容）

#### 2.3.2 核心挑战

根据讨论记录，**不同模型间无法直接复用 KV Cache**，原因包括：
- 架构差异（层数、维度、注意力头数不兼容）
- Tokenizer 不同
- 语义表示不一致

因此，必须在模型外部管理上下文状态，采用**文件系统作为持久化记忆**的策略。

#### 2.3.3 核心架构：三层记忆栈（Memory Stack）

将原来扁平的 MEMORY.md 拆分为三个语义层级，每层有不同的生命周期和 Token 预算：

```
┌─────────────────────────────────────────────────────┐
│               三层记忆栈（Memory Stack）              │
├─────────────────────────────────────────────────────┤
│  L1 战略层 │ ≤2000 tokens │ 跨会话持久 │ 极少更新   │
│  (Strategic Memory)                                 │
│  核心决策、架构约定、用户偏好、关键实体关系           │
├─────────────────────────────────────────────────────┤
│  L2 摘要链 │ ≤4000 tokens │ 按切换更新 │ 渐进合并   │
│  (Summary Chain)                                    │
│  每次切换生成增量摘要，历史摘要逐步合并压缩           │
├─────────────────────────────────────────────────────┤
│  L3 工作区 │ 动态分配     │ 当前会话   │ 切换时清理  │
│  (Working Memory)                                   │
│  最近N轮完整对话、当前plan/todo、正在编辑的代码片段    │
└─────────────────────────────────────────────────────┘
```

> ⚠️ Token 预算基于 128K-512K 上下文窗口设定，L1+L2 合计 ≤6000 tokens，仅占窗口的 1-5%。

##### （一）L1 — 战略层（Strategic Memory）

- **存储位置**：`MEMORY.md` 中的 `## 战略记忆` 区块
- **内容范围**：跨越整个会话生命周期的不可丢失信息
  - 用户核心偏好和约束
  - 重大技术决策及原因（简写形式）
  - 架构约定和设计原则
  - 关键实体关系图（简写）
  - `[PIN]` 标记的永不过期信息
- **Token 预算**：≤ 2000 tokens
- **更新频率**：仅在重大决策点写入
- **格式规范**：极度精简的键值对 / 简写形式

```markdown
## 战略记忆
- [PIN][架构] 采用微服务，网关=Kong，DB=PostgreSQL+Redis
- [PIN][约束] 用户要求所有API响应<200ms
- [决策-0405] 放弃方案A(Raw Context)，改用外部记忆系统
- [偏好] 代码风格=Google Style，语言=Python 3.11+
- [实体] UserService→AuthService→DB | OrderService→PaymentGateway
- [坑] MySQL连接池默认5太小→调到20，否则高并发超时
```

> `[PIN]` 标记的条目在任何合并/压缩操作中**永不被删除或进一步压缩**。

##### （二）L2 — 摘要链层（Summary Chain）

- **存储位置**：`MEMORY.md` 中的 `## 摘要链` 区块
- **内容范围**：按时间线组织的渐进式工作摘要
  - 每次模型切换时，由当前模型生成本阶段增量摘要
  - 历史摘要按 DAG 结构逐步合并压缩
  - 保留关键代码变更的文件路径 + 行号引用（非全文）
- **Token 预算**：≤ 4000 tokens
- **更新频率**：每次模型切换时
- **格式规范**：带时间戳和重要度的精简摘要条目

```markdown
## 摘要链
### [S3] 16:30 CODE_COMPLEX | imp=8 | Claude-Opus→Qwen-2.5-32B
完成 UserService 重构。修改: user_service.py:L45-120(认证逻辑重写)、
api_routes.py:L30-55(新增/v2/users端点)。待办: 单元测试未写。
上一段: S2

### [S2] 15:00 QA→Qwen-2.5-32B | imp=6 | 合并自[S1]
确认技术栈选择。讨论数据库索引策略，
决定对users表添加复合索引(email+created_at)。

### [S1] 14:00 (已合并入S2)
```

##### （三）L3 — 工作区层（Working Memory）

- **存储位置**：当前会话的上下文窗口（不持久化）
- **内容范围**：当前任务的活跃上下文
  - 最近 N 轮完整对话（默认 N=5）
  - 当前 `plan.md` 和 `todo.list` 的状态
  - 正在编辑的代码文件的相关片段
- **Token 预算**：动态分配（总窗口 - L1 - L2 - System Prompt）
- **生命周期**：随切换事件被提取、压缩或清理

#### 2.3.4 智能提取管道（Extraction Pipeline）

> 借鉴 ClawHub DR Context Pipeline 的确定性管道思想。在每次切换触发后、实际执行切换前，由**当前模型**执行 4 步提取流程。

```
切换触发（势能越过阈值）
  │
  ▼
[Step 1: 扫描] ──→ 扫描当前上下文窗口全部内容
  │
  ▼
[Step 2: 分类] ──→ 将内容分为 4 个桶:
  │                 🔴 关键(Key)：决策、架构、约束、错误教训
  │                 🟡 摘要(Sum)：任务进展、代码修改记录
  │                 ⚪ 冗余(Red)：寒暄、重复确认、中间调试过程
  │                 🔵 引用(Ref)：大段代码→仅保存路径+行号引用
  │
  ▼
[Step 3: 提取] ──→ 按分类执行不同策略:
  │                 🔴 Key → 简写后写入/更新 L1 战略层
  │                 🟡 Sum → 生成增量摘要，追加到 L2 摘要链
  │                 ⚪ Red → 直接丢弃（「瘦身」效果来源）
  │                 🔵 Ref → 记录文件路径+行号，附在摘要中
  │
  ▼
[Step 4: 校验] ──→ 检查提取结果完整性:
                    - L1 中所有 [PIN] 条目是否完好
                    - 关键实体和决策点是否被覆盖
                    - Token 预算是否在限额内
                    - 若校验失败 → 回退，不执行切换
```

##### 提取分类规则

| 信息类别 | 识别特征 | 处理策略 | 简写示例 |
| :--- | :--- | :--- | :--- |
| **重大决策** | "决定"、"采用"、"放弃"、"最终选择" | 提取决策+原因→L1 | `[决策] 用Redis缓存(非Memcached)，因需复杂数据结构` |
| **架构约定** | 模块关系、接口定义、数据流 | 提取核心关系→L1 | `Auth→JWT+RefreshToken, Gateway→Kong` |
| **错误教训** | "踩坑"、"注意"、"bug"、修复记录 | 保留根因+解法→L1 | `[坑] MySQL连接池默认5→调到20` |
| **用户偏好** | 明确的风格/约束要求 | 写入L1，去重 | `[偏好] 类型注解必须, docstring=Google格式` |
| **代码修改** | 文件变更、函数增删 | 路径+摘要→L2 | `修改: auth.py:L45-80(添加Token刷新)` |
| **任务进展** | TODO状态变化、里程碑完成 | 合并为进度快照→L2 | `完成3/7: ✅认证 ✅路由 ✅DB □测试...` |
| **寒暄/确认** | "好的"、"谢谢"、"明白了" | **丢弃** | — |
| **调试过程** | 中间尝试、错误尝试、反复修改 | **丢弃**（只保留最终方案） | — |
| **大段代码** | 超过 20 行的完整代码块 | 保存路径+行号引用→L2 | `参见: service.py:L10-85` |

#### 2.3.5 增量摘要引擎（Summary DAG）

构建树状摘要结构，避免信息随时间线性衰减。历史越久远，摘要越精炼，但关键信息始终可追溯。

##### DAG 结构示意

```
深度0（叶节点）    [S1] [S2] [S3] [S4] [S5] [S6] [S7] [S8] [S9]
                    \   |   /      \   |   /      \   |   /
深度1（合并）        [M1]           [M2]           [M3]
                       \            |             /
深度2（高层抽象）          [Root Summary]
```

##### 合并规则

| 规则 | 说明 |
| :--- | :--- |
| **触发条件** | 同一深度的摘要条目达到 **3 个**时，自动触发合并 |
| **压缩率** | 3 个条目合并后约为原总量的 **40%**（即压缩 60%） |
| **保留优先** | 合并时保留：关键决策、实体关系、最终结论、[PIN] 内容 |
| **丢弃优先** | 合并时丢弃：中间过程、已被后续决策覆盖的旧决策 |
| **[PIN] 保护** | 带 `[PIN]` 标记的条目在任何级别的合并中**不可删除或再压缩** |

##### 重要度衰减机制

每个摘要条目携带 `imp`（importance）分数，范围 1-10：

| 信息类型 | 初始 imp | 每次合并衰减 | 淘汰阈值 |
| :--- | :--- | :--- | :--- |
| 普通任务进展 | 5 | -1 | imp < 3 时可丢弃 |
| 重大决策 | 9 | -0.5 | imp < 3 时可丢弃 |
| `[PIN]` 标记 | 10 | **不衰减** | **永不淘汰** |

#### 2.3.6 切换重组编排器（Switch Composer）

切换执行时的核心编排逻辑，负责将三层记忆重新组装为新模型的初始上下文。

##### Token 预算分配

```
新模型上下文窗口（以 128K 为例）
├── System Prompt + Skill 指令       ~2000 tokens  (固定开销)
├── L1 战略记忆                       ≤2000 tokens  (全量注入)
├── L2 摘要链（最近 2-3 条 + Root）   ≤2500 tokens  (按时间优先)
├── 切换元信息（来源/原因/待办）       ~300 tokens   (固定)
└── 剩余给新对话                      ~121K tokens  (动态可用)
```

> 对于 512K 窗口的模型，L1/L2 预算可按比例放大，但总记忆开销不应超过窗口的 5%。

##### 注入格式（新模型收到的初始上下文）

```markdown
[系统] 模型切换上下文恢复

## 你的背景知识（战略记忆）
- [PIN][架构] 采用微服务，网关=Kong，DB=PostgreSQL+Redis
- [PIN][约束] 用户要求所有API响应<200ms
- [决策-0405] 放弃方案A(Raw Context)，改用外部记忆系统
- [偏好] 代码风格=Google Style，语言=Python 3.11+
- [实体] UserService→AuthService→DB | OrderService→PaymentGateway

## 最近工作摘要（摘要链）
### 上一阶段 [S3] 16:30
完成 UserService 重构。修改: user_service.py:L45-120(认证逻辑)、
api_routes.py:L30-55(新增/v2/users)。待办: 单元测试。

### 更早 [S2/M1] 15:00
确认技术栈(Python+FastAPI+PostgreSQL)。DB索引策略已定。

## 切换信息
- 来源模型: Claude-Opus-4 (权重 85)
- 目标模型: Qwen2.5-32B (权重 65)
- 切换原因: QA 任务权重差值 -20，连续 4 轮触发降级
- 当前待办: 编写 user_service.py 的单元测试
- 相关文件: user_service.py, api_routes.py, test_user.py
```

##### 切换执行流程（完整）

```
切换决策确认
    │
    ▼
[1. 提取管道] ──→ 执行 4 步提取（扫描→分类→提取→校验）
    │
    ├── 校验失败 → 中止切换，保持当前模型
    │
    └── 校验通过
         │
         ▼
[2. 写入 MEMORY.md] ──→ 更新 L1 战略层 + 追加 L2 摘要
         │
         ▼
[3. DAG 维护] ──→ 检查是否触发合并（同深度≥3）
    │                若触发 → 执行合并，更新 MEMORY.md
    │
    ▼
[4. 模型切换] ──→ 调用 session_status(model="目标模型")
    │
    ▼
[5. 上下文注入] ──→ 按预算分配策略组装初始上下文
    │                注入 L1 全量 + L2 最近条目 + 切换元信息
    │
    ▼
[6. 连续性验证] ──→ 新模型确认上下文已恢复
                     若恢复失败 → 触发回退机制
```

#### 2.4.5 命令错误处理

所有命令的错误输出遵循统一格式：

| 场景 | 错误信息示例 |
| :--- | :--- |
| 未知命令 | `❌ 未知命令: /ms foo。使用 /ms help 查看可用命令。` |
| 未知模型 | `❌ 未找到模型 "xxx"。使用 /ms list 查看可用模型。` |
| 参数缺失 | `❌ 缺少参数。用法: /ms set <模型> <任务缩写> <适配度>` |
| 权重越界 | `❌ 适配度必须在 0-100 之间，当前值: 150` |
| 导入文件不存在 | `❌ 文件不存在: config_backup.yaml` |
| 已处于该模式 | `⚠️ 当前已是自动模式，无需重复切换。` |
| L1 预算溢出 | `⚠️ PIN 失败：L1 战略层已达 Token 上限 (2000)。请先清理旧条目。` |

#### 2.4.6 实现方式

1. 创建 skill 文件，`description` 使用正则匹配 `/ms` 或 `/model-switch` 开头的消息
2. 实现命令解析器（Command Parser），支持子命令分发和参数校验
3. 模式切换：维护 `mode` 状态（auto / manual），持久化到会话存储
4. 基础切换：查询模型别名映射表，调用 `session_status(model="目标模型")` 执行切换
5. `router` / `set` / `reset` 命令：读取/修改 `routing_matrix.yaml` 文件
6. `debug` 命令：在回答后处理钩子中注入调试脚注
7. `stats` / `history`：从切换日志中聚合统计
8. `memory` / `pin`：读取/写入 MEMORY.md 文件的对应区块
9. `export` / `import`：序列化/反序列化全部配置文件
10. `cooldown` / `threshold`：修改运行时配置并持久化

```yaml

  # ══════════════════════════════════════════════════════════
  # 模型任务适配度矩阵（路由配置）
  # ══════════════════════════════════════════════════════════
  #
  # 安装流程：
  #   1. 读取 OpenClaw 的 openclaw.json 中已注册的可用模型列表
  #   2. 将每个模型 ID 与内置画像库 model_profiles.yaml 模糊匹配
  #   3. 匹配成功的模型继承画像权重，未匹配的默认权重为 default_weight
  #   4. 生成 routing_matrix.yaml 并输出给用户确认
  #   5. 用户可通过 /ms router 和 /ms set 命令随时查看和修改
  #
  # 路由逻辑：
  #   对于任务类型 T，在可用模型中按 routing_matrix[model][T] 降序排列，
  #   选取权重最高的模型作为推荐目标。
  #
  routing:
    matrix_file: "routing_matrix.yaml"      # 运行时适配度矩阵文件路径
    profile_library: "model_profiles.yaml"  # 内置画像库文件路径
    #
    # 📂 model_profiles.yaml 文件位置：
    #   <skill安装目录>/config/model_profiles.yaml
    #   例如: .openclaw/skills/auto-switch-skill/config/model_profiles.yaml
    #
    #   用户可直接编辑此文件来调整模型画像（能力权重、分类标签等），
    #   也可以让 Agent 通过文件操作工具来修改。
    #   修改后执行 /ms reset 使更改生效（重新生成路由矩阵）。
    #
    default_weight: 3                       # 未匹配画像的模型默认权重（0-100）
    weight_diff_threshold: 15               # 权重差值触发切换方向判定的最小阈值

  # 切换策略
  switcher:
    max_switches_per_session: 30    # 单会话最大切换次数（OpenClaw Agent 会话较长，需要更大上限）
    require_confirm_for_high_weight: false  # 高权重模型切换是否需要用户确认
    high_weight_confirm_threshold: 90      # 权重超过此值视为高权重模型

    # 切换势能（阻尼机制）
    damping:
      upgrade_increment: 40         # 升级方向（推荐权重>当前）每轮势能增量
      downgrade_increment: 20       # 降级方向（推荐权重<当前）每轮势能增量
      decay_rate: 30                # 一致/差值过小时每轮衰减量
      upgrade_threshold: 50         # 升级触发阈值（~2轮连续）
      downgrade_threshold: 80       # 降级触发阈值（~4轮连续）

    # 上下文惯性
    context_inertia:
      enabled: true
      threshold_multiplier: 1.5     # 惯性生效时阈值放大倍数
      topic_change_keywords:        # 话题转换信号词
        - "新的需求"
        - "另一个问题"
        - "换个话题"
        - "接下来"
      idle_reset_minutes: 30        # 超时重置惯性（分钟）

    # 冷却期
    cooldown_rounds: 3              # 切换后至少 N 轮不再切换

  # 上下文管理（三层记忆架构）
  context:
    strategy: "memory_stack"          # memory_stack(推荐) | memory_file(旧版) | raw_context

    # L1 战略层配置
    strategic_memory:
      max_tokens: 2000               # 战略层 Token 预算上限
      pin_enabled: true              # 启用 [PIN] 永不过期标记
      auto_dedup: true               # 自动去重（相同键的旧条目被新条目覆盖）

    # L2 摘要链配置
    summary_chain:
      max_tokens: 4000               # 摘要链 Token 预算上限
      merge_trigger: "depth_count"   # depth_count(同深度达N个合并) | token_overflow
      merge_threshold: 3             # 同一深度达到 3 个条目时触发合并
      compression_ratio: 0.4         # 合并压缩率（3合1后保留约40%内容）
      importance_decay:
        normal: 1.0                  # 普通进展每次合并衰减值
        decision: 0.5                # 重大决策每次合并衰减值
        pin: 0.0                     # [PIN] 条目不衰减
      importance_discard_threshold: 3 # imp 低于此值时可被丢弃

    # L3 工作区配置
    working_memory:
      preserve_recent_rounds: 5     # 保留最近 N 轮完整对话

    # 提取管道配置
    extraction_pipeline:
      executor: "current_model"      # current_model(推荐) | dedicated_model
      validation_enabled: true       # 提取后校验是否通过
      abort_on_validation_fail: true # 校验失败时中止切换

    # 切换重组配置
    switch_composer:
      inject_l1_full: true           # L1 战略层全量注入
      inject_l2_recent_count: 3      # L2 注入最近 N 条摘要
      inject_l2_root: true           # 同时注入 Root Summary（如有）
      max_memory_ratio: 0.05         # 记忆层占上下文窗口的最大比例
```

## 5. 技术风险与缓解措施

| 风险 | 影响 | 概率 | 缓解措施 |
| :--- | :--- | :--- | :--- |
| KV Cache 不可跨模型复用 | 切换后首次响应延迟高 | 确定 | 三层记忆栈 + 切换重组编排器注入关键上下文 |
| 任务评估准确率不足 | 错误的切换决策 | 中 | 阻尼系统（势能累积 + 惯性）+ 用户反馈校准 |
| 上下文信息丢失 | 对话连续性受损 | 低 | 4 步提取管道 + 校验回退 + [PIN] 永不过期保护 |
| 频繁切换导致体验碎片化 | 用户体验下降 | 中 | 冷却时间限制 + 切换次数上限 + 上下文惯性 |
| 提取管道分类不准确 | 关键信息被误分为冗余而丢弃 | 中 | 校验步骤兜底 + 保守分类策略（宁多勿少） |
| 摘要 DAG 过度压缩 | 历史信息失真 | 低 | 重要度衰减机制 + [PIN] 保护 + 压缩率上限 40% |
| `session_status` API 限制 | 切换方式受限 | 待验证 | 需要实际测试 API 能力边界 |

## 6. 开放问题

> 以下问题在需求评审时需要确认：

1. **`session_status` API 的能力边界？** 切换模型是否真的只需调用该接口？是否保留同一 session 的上下文？
2. **`weight_diff_threshold` 的最优值？** 当前设为 15，需要通过实际测试场景验证该阈值是否合理（太小会频繁触发势能累积，太大会对任务变化反应迟钝）。
3. **自动降级策略如何触发？** 复杂任务处理完毕后，如何判断何时降回轻量模型？
4. **多 Agent 场景下的切换协调？** 如果多个 Agent 同时触发升级切换，如何避免资源争抢？
5. **切换历史是否需要持久化？** 还是仅在内存中保持会话级别的记录？
6. **画像库更新机制？** 新模型发布后，`model_profiles.yaml` 的更新频率和流程如何？
