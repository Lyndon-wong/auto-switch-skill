# 模型切换 Skill 设计讨论记录

> **日期**：2026年4月5日
> **参与人**：主公（王林东）& 管理喵

---

## 一、核心需求与设计初衷

赋予 Agent "自知之明"。当 Agent 发现当前模型（如 Qwen-7B）处理复杂逻辑推理或超长文档感到吃力时，能主动触发 skill 切换到更强大的模型（如 GPT-4 或 Qwen-72B）；反之，在处理简单任务时切换回轻量模型以节省成本。

## 二、技术层面的阻碍：KV Cache 继承问题

**结论：** 目前无法直接在不同模型间复用 KV Cache。

- **架构差异**：不同模型的层数、隐藏层维度、注意力头数不兼容。
- **Tokenizer 不同**：即使是同一系列模型，其 Tokenizer 权重矩阵完全不同，Token 向量表示天差地别。
- **语义不一致**：不同模型对同一段文字的注意力权重分配（理解方式）不同。

## 三、解决方案对比

| 方案 | 优点 | 缺点 |
| :--- | :--- | :--- |
| **方案A：完全重新生成 (Raw Context)** | 逻辑简单，信息无损 | 计算量浪费，首次响应耗时（Prefill）随对话长度线性增长 |
| **方案B：摘要 + 压缩上下文** | 降低 Token 消耗，提高响应速度 | 可能丢失细节，不适合长程依赖任务（如代码修改） |
| **方案C：利用外部记忆系统 (MEMORY.md)** | 最稳健，符合 OpenClaw 文件导向设计 | 需要标准化写入格式 |
| **方案D：并行影子模型** | 零切换延迟 | 成本极高 |

## 四、重大发现：原生的动态切换机制

- **厂商机制（如 OpenAI）**：GPT-4o 与 GPT-4o-mini 可能存在嵌入共享（Embedding Sharing）或特征对齐，内部路由在某些中间层进行半透明切换。
- **vLLM 启示**：缓存虚拟化思想可用于加快新模型加载，关键在于"语义对齐"。

## 五、开发计划与工作量预估

1. **触发决策逻辑**：通过 Response 的 `thought` 字段解析困惑度、Token 报错率或显式的任务分类。
2. **状态保留**：切换前强制执行 `checkpoint`，将 `plan.md`、`todo.list` 及 `key_summary` 写入缓冲区。
3. **配置重载**：动态修改 Agent 的 `llm_config` 并重启推理实例。

**预估工作量：**

- 核心 Skill 框架：2-3 人日
- 上下文压缩算法：3-5 人日
- 测试与调优：约 1 周

## 六、特别发现：极简实现方案（`/ms` 风格）

讨论提到了一种极简的 `/ms` (model switch) 命令行切换方式：

- **特点**：不需要重建会话，直接在当前对话中切换，上下文可能保留（同一 session）。
- **实现三步走**：
  1. 创建 skill，description 使用正则匹配 `/ms` 开头的消息。
  2. 根据输入的命令解析模型别名（映射表）。
  3. **核心调用**：执行 `session_status(model="目标模型")` 即可完成切换。
